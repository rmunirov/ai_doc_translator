# Product Requirements Document: AI Doc Translator

## Executive Summary

**AI Doc Translator** — веб-приложение для перевода документов (PDF, TXT, HTML) с одного языка на другой с помощью LLM. Пользователь загружает документ через браузер, выбирает целевой язык, и получает переведённый файл с сохранённым форматированием.

Основное отличие от простых переводчиков — использование LangGraph-агента, который разбивает документ на чанки, сохраняет контекст между ними, применяет пользовательские глоссарии и восстанавливает исходное форматирование. Перевод выполняется асинхронно через очередь задач; пользователь видит статус в реальном времени.

MVP ориентирован на одного пользователя (без аутентификации) и поддерживает GigaChat как основной провайдер перевода.

---

## Mission

Сделать качественный машинный перевод профессиональных документов доступным без ручного разбиения на части, потери структуры и контекста.

**Ключевые принципы:**

1. **Format-first** — переведённый документ должен выглядеть как оригинал
2. **Context-aware** — перевод каждого фрагмента учитывает уже переведённое
3. **Terminology control** — пользователь управляет переводом ключевых терминов через глоссарий
4. **Async by default** — длинные документы переводятся в фоне без зависания UI
5. **Local-friendly** — поддержка локальных LLM (Ollama / LM Studio) наравне с облачными

---

## Target Users

**Основная аудитория MVP:** специалисты, работающие с технической и деловой документацией на нескольких языках — разработчики, аналитики, менеджеры.

**Потребности:**
- Перевести PDF/TXT/HTML без потери структуры (заголовки, списки, таблицы)
- Контролировать перевод терминологии (технические термины, названия продуктов)
- Видеть историю переводов и скачивать результаты повторно
- Работать с русским языком без деградации качества

**Технический уровень:** базовый — пользователь умеет загрузить файл и выбрать язык.

## MVP Scope

### ✅ В MVP (Core)

**Загрузка и перевод:**
- ✅ Загрузка документов: PDF, TXT, HTML (до 10 МБ)
- ✅ Выбор целевого языка из списка (RU, EN, DE, FR, ZH, ES)
- ✅ Автоопределение языка источника
- ✅ Асинхронный перевод с отображением прогресса (polling)
- ✅ Скачивание переведённого документа

**Качество перевода:**
- ✅ Чанкование документа перед переводом
- ✅ Передача контекста между чанками (скользящее окно)
- ✅ Сохранение форматирования (заголовки, списки, таблицы)
- ✅ Пользовательский глоссарий (термин → перевод)

**История и управление:**
- ✅ История переводов (список с датой, именем файла, языками)
- ✅ Повторное скачивание результата из истории
- ✅ Удаление записи из истории

**Провайдеры LLM:**
- ✅ GigaChat (основной)
- ✅ Локальная модель через Ollama (настраивается в .env)

### ❌ Вне MVP

- ❌ Аутентификация и мультипользовательский режим
- ❌ DOCX, PPTX, XLSX форматы
- ❌ Пакетная загрузка нескольких файлов
- ❌ WebSocket для прогресса (достаточно polling)
- ❌ Редактирование перевода в браузере
- ❌ Сравнение оригинала и перевода side-by-side
- ❌ Публичный REST API для внешних клиентов
- ❌ Docker / деплой (определяется позже)
- ❌ Уведомления по email о готовности перевода
- ❌ Выбор модели через UI (только через .env)

## User Stories

**US-1: Перевести PDF-документ**
- Как пользователь, я хочу загрузить PDF и получить его перевод на русский язык, чтобы прочитать иностранный технический документ.
- *Пример:* Загружаю английский whitepaper (15 страниц), выбираю RU, через 2 минуты скачиваю переведённый файл с сохранёнными заголовками и таблицами.

**US-2: Контролировать перевод терминов**
- Как пользователь, я хочу задать глоссарий (например, "API Gateway → Шлюз API"), чтобы технические термины переводились предсказуемо.
- *Пример:* Добавляю в глоссарий 10 терминов, запускаю перевод — в результате все они переведены именно так, как я указал.

**US-3: Видеть прогресс перевода**
- Как пользователь, я хочу видеть, что перевод идёт, а не просто крутящийся спиннер, чтобы понимать, сколько ещё ждать.
- *Пример:* Страница показывает "Переведено 4/12 фрагментов" и обновляется каждые 3 секунды.

**US-4: Вернуться к старому переводу**
- Как пользователь, я хочу найти перевод, сделанный неделю назад, и скачать его снова, чтобы не переводить повторно.
- *Пример:* В разделе "История" нахожу нужный файл по дате и имени, нажимаю "Скачать".

**US-5: Перевести HTML-страницу**
- Как пользователь, я хочу вставить HTML-файл и получить переведённый HTML с сохранёнными тегами, чтобы встроить его в сайт.
- *Пример:* Загружаю HTML-страницу с `<h1>`, `<p>`, `<ul>` — в результате теги сохранены, переведён только текстовый контент внутри них.

## Core Architecture & Patterns

### Высокоуровневая архитектура

```
Browser (Jinja2 UI)
    │  multipart/form-data upload
    ▼
FastAPI
    ├── GET /           → dashboard (Jinja2)
    ├── POST /translate → enqueue job, return job_id
    ├── GET /status/{id}→ job status + progress
    ├── GET /download/{id} → result file
    ├── GET /history    → list of past jobs
    └── /glossary       → CRUD for user glossary
         │
         ▼
   AsyncIO Task Queue (asyncio + background tasks)
         │
         ▼
   LangGraph Agent
    ├── [node] detect_language
    ├── [node] parse_document     → extract text + structure
    ├── [node] chunk_document     → overlapping chunks
    ├── [node] translate_chunk    → GigaChat / Ollama + glossary injection
    ├── [node] update_context     → rolling summary for next chunk
    └── [node] assemble_document  → reconstruct with original format
         │
         ▼
   PostgreSQL
    ├── jobs (id, status, progress, created_at, result_path)
    ├── history (job metadata for UI)
    └── glossary (term, translation, user_id)
```

### Паттерны

- **LangGraph state machine** — каждый шаг перевода — отдельная нода; состояние (State) передаётся между нодами явно через TypedDict
- **Repository pattern** — все SQL-запросы инкапсулированы в `app/models/repositories/`
- **Dependency Injection** — FastAPI `Depends()` для DB-сессий, LLM-клиентов
- **Background Tasks** — FastAPI `BackgroundTasks` для запуска агента без блокировки

- ❌ HTTPS (локальный запуск)

## API Specification

### POST /translate

Загрузить документ и запустить перевод.

**Request:** `multipart/form-data`
```
file:        <binary>
target_lang: "ru" | "en" | "de" | "fr" | "zh" | "es"
```

**Response `202`:**
```json
{ "job_id": "uuid", "status": "queued" }
```

---

### GET /status/{job_id}

**Response `200`:**
```json
{
  "job_id": "uuid",
  "status": "processing",
  "progress": { "done": 4, "total": 12 },
  "created_at": "2026-02-24T10:00:00Z"
}
```
`status`: `queued | processing | done | failed`

---

### GET /download/{job_id}

Возвращает переведённый файл (`Content-Disposition: attachment`).

---

### GET /history

```json
{
  "items": [
    {
      "job_id": "uuid",
      "filename": "report.pdf",
      "source_lang": "en",
      "target_lang": "ru",
      "status": "done",
      "created_at": "2026-02-24T10:00:00Z"
    }
  ]
}
```

---

### GET/POST/DELETE /glossary

**GET** — список всех терминов.
**POST body:**
```json
{ "term": "API Gateway", "translation": "Шлюз API" }
```
**DELETE /glossary/{term_id}** — удалить термин.

## Risks & Mitigations

**Risk 1: Качество перевода GigaChat деградирует на длинных документах**
- *Mitigation:* настраиваемый размер чанка + overlap; в промпт передаётся краткое резюме предыдущих чанков.

**Risk 2: GigaChat API недоступен / rate limit**
- *Mitigation:* retry с exponential backoff (tenacity); fallback на Ollama если настроен.

**Risk 3: PDF с нестандартной структурой (сканы, колонки)**
- *Mitigation:* в MVP явно документируем ограничения; сканы (без текстового слоя) выдают понятную ошибку.

**Risk 4: Большие файлы перегружают память**
- *Mitigation:* лимит 10 МБ на загрузку; потоковая обработка чанков без загрузки всего документа в RAM.

**Risk 5: Потеря задачи при перезапуске приложения**
- *Mitigation:* статус задачи хранится в PostgreSQL; при старте приложение переводит `processing` → `failed` для "подвисших" задач.
